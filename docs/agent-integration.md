# Closed-loop mathematical reasoning with Arithma

This guide is for the engineer wiring an AI agent to Arithma's MCP server, and
for the agent itself. It documents the *loop* — attempt a step, verify it,
branch on the evidence, correct or proceed — with worked transcripts, every one
of which was generated by the binary built from this tree (the build commit is
stamped in each response; see [Provenance](#provenance)).

It is the longer companion to the [`using-arithma`
skill](../.claude/skills/using-arithma/SKILL.md). The skill is the terse
in-context ruleset ("delegate every computation; switch on fields, not prose").
This document is the *why and how*, with transcripts you can replay. The
evidence taxonomy it consumes is specified in full in
[`result-status.md`](result-status.md); read that for the field-level contract.
A fully worked application built on this loop is
[`gum-beam-uncertainty.md`](../examples/gum-beam-uncertainty.md).

## Why a loop changes the class of problem

An agent that generates a long derivation open-loop — no checking — fails
almost surely. Suppose each step is right 95% of the time and errors are
independent:

| Regime | Model | P(correct 100-step result) |
|---|---|---|
| Open loop (no checking) | $0.95^{100}$ | **0.6%** |
| Closed loop, verification catches 95% of errors | $(1 - 0.05\times0.05)^{100}$ | **78%** |

(The closed-loop figure assumes a caught error is corrected and re-derived; the
per-step undetected-error rate is 5% of steps × 5% escaping the check = 0.25%.
The model is illustrative — real error correlation is not independent — but the
shape is the point: **verification is what makes multi-step reasoning viable at
all.** It is not polish applied at the end. It is the mechanism that keeps the
error probability from compounding to certainty.)

The corollary sets the default pattern: **check each step as you make it**, not
just the chain at the end. Per-step checking catches an error while it is one
step old and localizable, before it is buried under ten more steps that inherited
it. This is the same finding as the process-vs-outcome supervision result in the
reasoning literature: supervising each step beats grading only the answer.

## The loop

```
   ┌─────────────────────────────────────────────────────────┐
   │  1. Generate the next step (your algebra)                │
   │  2. verify_chain([prev_accepted, new_step])   ← 2 steps  │
   │  3. switch on verdict:                                   │
   │       pass  → accept step; goto 1                        │
   │       fail  → read first_failure + counterexample;      │
   │               the counterexample IS the diagnosis;      │
   │               correct the step; goto 2                   │
   │       inconclusive → the check couldn't decide; supply  │
   │               an exact value / different route; goto 2   │
   │  4. when the derivation is complete, run the FULL chain  │
   │     once more; read weakest_step for the evidence floor  │
   └─────────────────────────────────────────────────────────┘
```

Three rules make this work, and each is illustrated with a real transcript below:

1. **Switch on fields, never on prose.** Every verdict-shaped response carries a
   machine `verdict` (`pass` / `fail` / `inconclusive`) and a `status` (the six
   evidence tiers). Prose is reworded freely and the *same* word appears in a
   confirmation and a refutation. Read `verdict`, `status`, `mechanism`,
   `first_failure`, `weakest_step`. Never regex the sentence.
2. **The counterexample is the diagnosis — feed it back verbatim.** On a `fail`,
   the response hands you a specific point and both exact values. Do not invent a
   repair; read what the numbers say and fix *that*.
3. **A refusal is an answer.** `provably_impossible` is a theorem and
   `unable_to_compute` is honest ignorance. Neither is retried verbatim. One is
   reported; the other is routed around.

## The default call: an incremental two-step chain

To check one new step against the last accepted one, send a two-step
`verify_chain`: the previous (accepted) expression as the **anchor** (step 0,
which asserts nothing), and the new step declaring its `relation` to it.

```json
{"name": "verify_chain", "arguments": {"steps": [
  {"label": "previous accepted step", "expr": "..."},
  {"label": "new step",               "expr": "...", "relation": "equals"}
]}}
```

The `relation` names how the new step is supposed to follow from the anchor.
Each relation earns evidence by a different mechanism; the ones you will use most:

| Relation | The new step is… | Can reach `exact`? |
|---|---|---|
| `equals` | algebraically equal to the previous | yes, inside the ℚ(x) fragment |
| `derivative_of` | the derivative of the previous w.r.t. `variable` | yes |
| `integral_of` | an antiderivative of the previous | yes (round-trip is algebraic) |
| `substitution` | the previous with `variable` := `value` | yes |
| `solution_of` | a claimed root of the previous (an equation) | yes, for roots in the fragment |
| `implies` | implied by the previous | **no — capped at `verified`** |

The full relation table, with the exact mechanism each uses and why `implies`
and equation-`equals` cannot reach `exact`, is in
[`result-status.md`](result-status.md#verify_chain).

**When to use which tool.** Use `verify_chain` for a *derivation* — anything
where step $k$ is supposed to follow from step $k-1$. Use a single `equivalent`
or `verify` call for a standalone "are these two expressions equal?" with no
chain context. Use `equivalent` (not `verify`) when you need the possibility of
a *proof*: `equivalent` can return `exact`; `verify` is numeric by definition and
caps at `verified`. The `verify_chain` per-step machinery is the same ladder
`equivalent` uses, plus the relation types and the chain-level `first_failure` /
`weakest_step` bookkeeping.

## Transcript 1 — catching and correcting a mid-chain error

An agent differentiating $f(x) = (x^2+1)^3$ by the chain rule produces
$f'(x) = 3(x^2+1)^2 \cdot 2x$ but drops the inner factor of 2, claiming
$3x(x^2+1)^2$. It checks the step before proceeding:

```json
{"name": "verify_chain", "arguments": {"steps": [
  {"label": "f",                        "expr": "(x^2+1)^3"},
  {"label": "f prime (wrong coeff 3x)", "expr": "3x(x^2+1)^2",
   "relation": "derivative_of", "variable": "x"}
]}}
```

```
Chain: FAIL at step 1 "f prime (wrong coeff 3x)"
  0. f — anchor
  1. f prime (wrong coeff 3x) [derivative_of] — fail (exact — disagreement
     established in exact rational arithmetic — a disproof, not a tolerance
     judgement; derivative_rules+exact_rational_sample)
     counterexample: {"lhs":4.6875,"lhs_exact":"75/16","point":{"x":0.5},
                      "rhs":2.34375,"rhs_exact":"75/32"}
```

Read the fields, not the sentence:

- `verdict: "fail"`, `first_failure: 1` — the bug is at step 1.
- `status: "exact"` with caveat `exact_disagreement` — this is a **disproof**,
  not a tolerance wobble. The two sides differ in exact rational arithmetic. A
  provably-false step is refuted, never tolerated.
- The counterexample carries `lhs_exact` and `rhs_exact` — two *exact* rationals,
  not just f64 renderings (two distinct rationals can share an f64 image, so the
  exact strings are the ground truth). The raw fields are LaTeX strings
  (`"lhs_exact":"\\frac{75}{16}"`), rendered here and below as `75/16` for
  readability. At $x=\tfrac12$: the independently re-derived value is
  $\tfrac{75}{16}$; the claimed expression evaluates to $\tfrac{75}{32}$.
  **The claim is exactly half the truth.** The ratio is $2$ —
  which points straight at the missing factor of 2. The counterexample did not
  just say "wrong"; it said *how* wrong, and the arithmetic names the fix.

The agent corrects the coefficient to $6x$ and re-checks:

```json
{"name": "verify_chain", "arguments": {"steps": [
  {"label": "f",           "expr": "(x^2+1)^3"},
  {"label": "f prime (6x)", "expr": "6x(x^2+1)^2",
   "relation": "derivative_of", "variable": "x"}
]}}
```

```
Chain: PASS (2 steps; weakest evidence: exact at step 1 "f prime (6x)")
  0. f — anchor
  1. f prime (6x) [derivative_of] — pass (exact; derivative_rules+canonical_form_Q)
```

`verdict: "pass"`, `status: "exact"`, mechanism `derivative_rules+canonical_form_Q`
— the derivative rules ran and the resulting equality was decided by
canonical-form comparison over ℚ. This step is *proven*, and the agent proceeds
with the error corrected before it could propagate.

**Do not generatively repair.** When you see the `fail`, do not ask the model to
"try a different derivative." Read the counterexample, see the factor of 2, fix
the coefficient. The whole value of the loop is that the correction is driven by
the arithmetic, not by another sample from the same distribution that produced
the error.

### The same, at chain length: `first_failure` locates the bug

In a longer derivation, `first_failure` is the line number of the diagnosis.
An agent expands $(x+1)^2$ correctly, then makes a transcription slip:

```json
{"name": "verify_chain", "arguments": {"steps": [
  {"label": "start",   "expr": "(x+1)^2"},
  {"label": "expand",  "expr": "x^2+2x+1", "relation": "equals"},
  {"label": "typo +2", "expr": "x^2+2x+2", "relation": "equals"}
]}}
```

```
Chain: FAIL at step 2 "typo +2"
  0. start — anchor
  1. expand [equals] — pass (exact; difference_zero_Q)
  2. typo +2 [equals] — fail (exact — disagreement ...; exact_rational_sample)
     counterexample: {"lhs":2.25,"lhs_exact":"9/4","point":{"x":0.5},
                      "rhs":3.25,"rhs_exact":"13/4"}
```

The chain-level status reports `first_failure: 2` and `weakest_step: 1`. **These
differ, and the difference matters.** `weakest_step` marks the *evidence floor*
among the steps (step 1, whose passing evidence is the weakest non-failing rung);
`first_failure` marks the *bug*. When you are explaining a failure to correct it,
**follow `first_failure`, never `weakest_step`** — the diagnosis is at step 2, and
the passing step 1 must not outrank it. (The contract states this rule explicitly;
here is a live instance of it.)

## The escalation contract — `weakest_step` as a router

A chain can pass yet not be *proven*. Transcendental steps are the usual case:
they are checked numerically and cap at `verified` — evidence at $n$ points,
never a proof. An agent applies the double-angle identity, a true statement:

```json
{"name": "verify_chain", "arguments": {"steps": [
  {"label": "double angle", "expr": "\\sin(2x)"},
  {"label": "expand",       "expr": "2\\sin(x)\\cos(x)", "relation": "equals"}
]}}
```

```
[verified] numeric evidence, 12 points — not proof
Chain: PASS (2 steps; weakest evidence: verified at 12 points at step 1 "expand")
  0. double angle — anchor
  1. expand [equals] — pass (verified at 12 points; numeric_sample)
```

`verdict: "pass"` but `status: "verified"`, `weakest_step: 1`, mechanism
`numeric_sample`. The identity is true — but the tool checked it at 12 points and
says so; it did not prove it. **This is the router.** `weakest_step` names the
rung that determines how much you may claim, and the status tells you which
escalation move (if any) the downstream use requires:

| You have | You may claim | If you need more, escalate by… |
|---|---|---|
| `exact` | proven (a ℚ(x)-identity) | nothing — you are done |
| `verified` | evidence at $n$ points | **re-derive** by a different route; or hand the step to a **proof assistant** (e.g. Lean) for a formal proof. *More sample points will not help* — no amount of point-agreement upgrades `verified` to `exact`. |
| `heuristic` | believed, unchecked | get an independent check before building on it |
| `inconclusive` verdict | nothing yet | supply an exact value, split the step, or change representation |

The discipline: **do not treat `verified` as `exact`.** If the final result only
needs to be *probably* right for a downstream numeric use, `verified` is fine —
stop. If it will be a lemma other proofs depend on, `weakest_step` has told you
exactly which rung is still a promissory note, and the escalation is to prove
that one step, not to re-run the sampler.

## Transcript 2 — impossibility as a strategic redirect

A refusal that is a *theorem* is not a failure to route around — it is
information that redirects strategy. An agent tries to integrate a Gaussian:

```json
{"name": "integrate", "arguments": {"expr": "e^{-x^2}", "variable": "x"}}
```

```
[provably impossible] This integral has no formula using elementary functions
(polynomials, exponentials, logarithms, trigonometric). This is a theorem, not a
limitation of the tool. — antiderivative in special functions:
\frac{\sqrt{\pi}}{2} \cdot \erf(x)
```

```json
"result_status": {
  "status": "provably_impossible",
  "proof_certificate": {
    "method": "risch-de",
    "reason": "No elementary antiderivative exists. The differential equation
               q' + (-2x)·q = 1 has no rational solution.",
    "explanation": "This integral has no formula using elementary functions..."
  },
  "special_function": "erf",
  "special_form": "\\frac{\\sqrt{\\pi}}{2} \\cdot \\erf(x)"
}
```

Read `status: "provably_impossible"` and stop trying to find an elementary
antiderivative — **no rephrasing of the prompt will succeed, because a theorem
says it cannot.** `proof_certificate.method` (`risch-de`) tells you *which*
theorem. Then note the fork:

- **Redirect A — accept the recognized closed form.** `special_function: "erf"`
  and `special_form` give you the answer in the extended function system:
  $\int e^{-x^2}\,dx = \tfrac{\sqrt\pi}{2}\,\mathrm{erf}(x)$. Report *that*. (The
  recognition is guarded by a differentiation round-trip; an unrecognized
  integrand gets the bare certificate and no guessed name.) Note that `erf` is
  symbolic-only — it parses, prints, and differentiates, but numeric evaluation
  refuses; if you need a number, take Redirect B.

- **Redirect B — change representation.** For a *definite* value like
  $\int_0^1 e^{-x^2}\,dx$ (the definite call also returns the theorem — the tool
  will not fabricate a number), switch to a series whose terms *are* elementary:

  ```json
  {"name": "taylor_series", "arguments": {"expr": "e^{-x^2}", "variable": "x", "order": 6}}
  ```
  ```
  -\frac{1}{6} \cdot x^{6} + \frac{1}{2} \cdot x^{4} - x^{2} + 1        [exact]
     caveat: truncation — equality holds as series expansion, not as identity
  ```
  ```json
  {"name": "integrate",
   "arguments": {"expr": "1 - x^2 + \\frac{x^4}{2} - \\frac{x^6}{6}", "variable": "x"}}
  ```
  ```
  C - (\frac{1}{3} \cdot x^{3} - (-\frac{1}{42} \cdot x^{7} + \frac{1}{10} \cdot x^{5}) - x)     [exact]
  ```

  Each redirect step still carries its evidence: the Taylor expansion is `exact`
  with an honest `truncation` caveat (it is the series, not the function), and the
  term-by-term integral is `exact` by differentiation round-trip. The impossibility
  did not end the derivation — it forked it, and the new branch is verified at
  every step.

Impossibility also arrives from `solve`. A quadratic with negative discriminant:

```json
{"name": "solve", "arguments": {"equation": "x^2+x+1=0", "variable": "x"}}
```

```
[provably impossible] This equation has no real solutions. Both roots are complex.
This is a theorem (negative discriminant), not a limitation of the tool.
```

`method: "negative-discriminant"`. The redirect: if the model lives over the
reals, report "no solution" as the *result* and move on; if complex roots are
admissible, the agent knows to change domain rather than re-running `solve`
against the same wall. Either way, the theorem is the fork, and retrying the
identical call is the one move guaranteed to waste a turn.

Contrast with `unable_to_compute`, which is honest ignorance, not a theorem:
route around it (different technique, split the problem, supply more structure),
but do **not** fill the gap with a guessed answer — that discards exactly the
information the refusal carried.

## Honest limits — build the loop knowing these

- **`verified` is not proof.** It is agreement at $n$ points. Never say "proven"
  on a `verified` result. One subtle tell: if a `verified` equality reports a
  *nonzero* symbolic `Difference` while the points agree, the two sides differ
  below f64 tolerance — treat the step as open and cite the difference, don't
  round it away.
- **Equality is ℚ(x)-equality.** `exact` on an `equals` step means equality in the
  rational function field: $(x^2-1)/(x-1) = x+1$ holds, and removable domain
  points (here $x=1$) do not count as disagreement. If your application cares
  about pointwise equality of partial functions, that is a *different* notion the
  tool does not currently decide.
- **f64 boundaries are real.** No complex numbers, no relational assumptions
  ($x < y$), improper integrals refused. In a long derivative cascade,
  intermediate expressions grow geometrically — **substitute numeric values
  early**, as soon as a variable can be soundly pinned, to keep expressions small.
  A call that does not return promptly is *stalled, not thinking*: kill it,
  reorder or split the step, and treat it as `unable_to_compute` with the reason.
- **The input boundary is strict, and that is a feature.** A malformed argument
  is rejected with a JSON-RPC `-32602` naming the offending field, never silently
  ignored:

  ```json
  {"name": "solve", "arguments": {"expr": "x^2+1=0", "variable": "x"}}
  ```
  ```
  error -32602: Invalid arguments for tool 'solve': arguments: missing required
  field 'equation'; unknown field 'expr' (allowed: assumptions, equation, variable)
  ```
  When you see a `-32602`, fix the call — you named a field wrong (here `expr`
  should be `equation`). The tool will not guess what you meant, because guessing
  is how a wrong answer wears a confident face.

## Delivery surface — where the evidence actually reaches your agent

The evidence object rides in three places, and **which one your host forwards to
the model determines what you can switch on:**

- `result_status` as a **sibling of `content`** — raw-JSON consumers see it here.
- `result_status` **inside `structuredContent`** — typed MCP SDKs drop unknown
  top-level fields, so for those hosts this is the *only* surface that arrives.
  It carries the identical object.
- **In-band in `content[0].text`** — a text marker, because many MCP hosts deliver
  *only* `content.text` to the model. Loud statuses get a marker line
  (`[verified] …`, `[provably impossible] …`); `equivalent` and `verify` carry
  their tier in the sentence itself (`Equivalent: true [exact]` vs
  `Equivalent: likely true [verified at 12 points — numeric agreement, not
  proof]`). This exists so the tier survives even a text-only host — a contract
  is only as good as its delivery surface.

**Wiring guidance for the integrator:** if your host forwards `structuredContent`
or the raw result, switch on `result_status.status` / `.verdict` /
`.first_failure` there — it is machine-typed and unambiguous. If your host is
text-only, parse the marker line (it is stable) and the bracketed tier. Do not
parse the free prose. If you control the host, forward `structuredContent`.

## Provenance

Every response carries a `build` stamp — `{"commit": "...", "dirty": false}` —
so any transcript is self-dating and reproducible. The transcripts here were
generated by the binary built from this tree at commit `f5035a3`; the source is
unchanged from there through this doc's commit, so a build at any of them
reproduces them (your build stamp reports whichever commit you built).

**Rendering convention.** Transcript blocks show the tool's *real* output, with
two disclosed readability edits and nothing else: (1) a bracketed tier such as
`[exact]` or `[verified]` summarizes the response's `result_status`, which
travels in a separate field, not in `content.text`; (2) in counterexample
objects, the exact-value fields — LaTeX strings like `\frac{75}{16}` — are shown
as `75/16`. Everything else, including expression output like
`\frac{1}{6} \cdot x^{6}`, is verbatim. Replaying a call reproduces the raw form;
the mathematics is identical either way.

Each call is a single JSON-RPC line to the `arithma-mcp` binary on stdin; to
reproduce any block:

```sh
printf '%s\n' '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"verify_chain","arguments":{"steps":[{"label":"f","expr":"(x^2+1)^3"},{"label":"f prime (6x)","expr":"6x(x^2+1)^2","relation":"derivative_of","variable":"x"}]}}}' | arithma-mcp
```

A `dirty: true` stamp means the binary was built from an uncommitted tree — treat
such a transcript as provisional until the tree is committed.

## For the infrastructure author

The acceptance test for this pattern is a real agent prototype driving it, and
the friction reports that come back. The two knobs most likely to matter in
practice: (1) the **delivery surface** above — a host that swallows
`structuredContent` and truncates `content.text` will starve the agent of exactly
the evidence fields that make the loop work; forward the structured payload. (2)
**Per-step vs end-of-chain checking** — the incremental two-step chain is the
default because it localizes the bug to one step, but there is a per-call cost;
for a very long derivation, a reasonable middle path is per-step checks during
generation plus one full-chain check at the end (the full chain's `weakest_step`
is the audit floor for the whole result). If a prototype surfaces a case where
the field contract is awkward to consume, that is a defect in this contract, not
in the prototype — report it.
